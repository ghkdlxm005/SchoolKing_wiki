# LESSON-0004 — 툴박스 에셋은 넣기 전에 검사한다

## 규칙

1. 무료(툴박스) 모델을 넣으면 반드시 스크립트 검사를 돌린다.

   ```lua
   require(game.ServerStorage.SecurityScan)()
   ```

2. "복사해서 커맨드 바에 붙여넣으라"는 안내는 무조건 공격이다. 로블록스는 그런 요구를 하지 않는다. 커맨드 바는 스튜디오 최고 권한이라, 실행하는 순간 게임에 무엇이든 심을 수 있다.

3. 내용을 읽지 않은 코드는 실행하지 않는다. 자신이 직접 확인한 코드를 커맨드 바에서 돌리는 건 문제없다. 위험한 것은 출처와 내용을 모르는 코드다.

4. 의심스러우면 "파트가 0개인 모델"을 먼저 본다. 눈에 보이는 오브젝트가 하나도 없는데 스크립트만 들어 있는 모델은, 꾸미기가 아니라 코드를 심으려고 들어온 것이다.

## 위험 신호 (하나라도 있으면 열어볼 것)

| 신호 | 의미 |
| --- | --- |
| `require(숫자)` | 외부 에셋 코드를 받아 실행. 주인이 언제든 내용을 바꿀 수 있다 |
| `GetObjects` / `InsertService` | 외부 에셋을 게임에 삽입 |
| `loadstring` / `HttpGet` | 문자열 실행 / 외부 통신 |
| `script.Parent = nil` | 자기 은폐. 실행은 계속되면서 탐색기에서 사라진다 |
| `SetCoreGuiEnabled(..., false)` | 로블록스 기본 UI 차단(메뉴·채팅) |
| 클래스와 이름 불일치 | `Script`인데 이름이 `PackageLink`·`SurfaceWeld`·`Weld` 등 |
| 스크립트에 붙은 긴 문자열 속성 | 페이로드 은닉. 역순·인코딩된 코드일 수 있다 |

## 왜 (출처)

무료 모델에 섞여 들어온 백도어가 가짜 "Error 501" 창을 띄워, 커맨드 바에 명령어를 붙여넣도록 유도했다. 해독된 페이로드는 공격자 스크립트를 `ServerScriptService`에 설치하는 코드였다.

특히 다음 두 가지가 일반적인 코드 검색을 무력화했다.

- 문구와 명령어를 소스가 아니라 속성에 숨김 → `Error 501` 문자열 검색 실패
- 에셋 ID를 속성에 저장 후 require → `require(%d+)` 검색 실패

소스 grep만으로는 못 잡는다. 속성까지 함께 검사해야 한다.

→ [BUG-0008](../bugs/BUG-0008-toolbox-backdoor.md)

## 적용 대상

- 툴박스에서 가져오는 모든 모델·플러그인
- 특히 장식용 모델(조명·가구·NPC) — 코드가 있을 이유가 없는데 스크립트가 붙어 있으면 의심
- 협업자가 넣은 에셋도 동일하게 검사
