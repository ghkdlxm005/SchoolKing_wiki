# LESSON-0003 — 버프 값은 단일 함수로만 적용한다

## 규칙

이동속도처럼 **외부 효과(버프/디버프)가 얹히는 값**은 두 가지를 지킨다.

1. **값 변경 자체에 반응한다.** 상태 전환(state change) 시점에만 반영하면, 상태가 그대로일 때 들어온 효과는 적용되지 않는다.
2. **최종 값은 한 함수에서만 계산한다.** 그 값을 쓰는 모든 지점이 그 함수를 경유해야 한다. 어딘가에서 기본값을 직접 대입하면 그 경로로 빠져나갈 때 버프가 사라진다.

```lua
-- 기본 속도 + 외부 보정을 한 곳에서 계산
local function targetSpeedFor(state)
    return math.max(1, baseSpeedForState(state) + (Character:GetAttribute("SpeedBonus") or 0))
end

-- 값이 바뀌면 상태 전환이 없어도 즉시 반영
Character:GetAttributeChangedSignal("SpeedBonus"):Connect(function()
    setHumanoidSpeed(targetSpeedFor(currentState), 0.25)
end)
```

또한 Lua에서 **핸들러는 정의 시점의 상위 스코프를 캡처**한다. 계산 함수보다 위에서 핸들러를 등록하면 그 함수는 `nil`이 되어 호출 시 조용히 실패한다. 등록은 반드시 **정의 이후**에 한다.

## 왜 (출처)

Monster Energy 이동속도 버프 구현 중 같은 성격의 문제가 **두 번** 발생했다.

- 속성 변경 핸들러를 계산 함수보다 **125행 앞**에 등록해, 핸들러 안에서 함수가 `nil`이었다. 값이 바뀔 때마다 핸들러가 죽고, `setState`가 불리는 **상태 전환 순간에만** 버프가 반영됐다. 증상은 "상태가 안 바뀌면 이속이 적용되지 않는다".
- 슬라이드 종료·슬라이드 점프·맨틀·벽 점프·비정상 상태 복구·힐 종료 등 **7곳**에서 `Humanoid.WalkSpeed`에 기본값을 직접 대입해, 그 동작을 하면 버프가 유실됐다. 증상은 "가끔 이속 증가가 안 먹는다".

## 적용 대상

- 이동속도(`SpeedBonus`), 시야각(`FovBonus`) 등 **누적·합산되는 보정값** 전반
- 향후 추가될 버프/디버프 아이템 — 새 값을 도입할 때 "계산 함수 1개 + 변경 시 재적용 1개"를 먼저 만든다
- 값을 대입하는 지점을 늘릴 때는, 기본값이 아니라 **계산 함수를 호출하는지** 확인한다
